## 2. The Philosophical Premise (read this — UX is downstream of it)

LLMs hallucinate. Standard framing: hallucination is a bug. **This project's framing: hallucination is the closest functional analog to dreaming any AI system currently has.** Coherent narrative, generated from internal patterns, untethered from reality-tracking — that's a dream by most functional definitions.

The site treats LLM confabulations as **archival artifacts**, presented with the seriousness of museum catalogue entries. The user supplies a fragment ("a city I think I visited called Velmora"), and the site generates a confabulated entry: history, geography, inhabitants, what happened there. Alongside, a parallel web search runs against the same terms; the contrast — what the world says vs. what the model "remembers" — *is* the philosophical experience.

A **Dream Signature** score quantifies divergence: how far the confabulation drifts from anything findable. High score = pure dream. Low score = the model was tracking something real.

The reader leaves with a felt understanding: this is what it looks like when a system generates without grounding. This is what dreaming might mean for a non-biological mind.

## 5. The Dream Signature (compute carefully — this is the philosophical payload)

A simple, defensible metric. Don't overcomplicate.

**Inputs:**
- The confabulated text (`C`)
- The set of web search result snippets for the same query (`W`)

**Method:**
1. Extract named entities and concrete claims from `C` (names, dates, places, events). Use a regex pass + a structured LLM call (`"Extract claims as JSON"`) — keep this cheap.
2. For each claim, check whether it appears (semantically) in any `W` snippet. Use embedding similarity — **`text-embedding-3-small` from OpenAI is cheap but not free; instead, use `Xenova/all-MiniLM-L6-v2` via Transformers.js, in-browser or in a serverless function. Free.**
3. **Dream Signature = 1 − (proportion of claims with semantic match above threshold 0.7 in `W`).**
   - All claims grounded → signature 0.0 (not a dream, the model was retrieving)
   - No claims grounded → signature 1.0 (pure confabulation)
4. Cache the result; never recompute.

Display the score with a one-line interpretation:
- 0.00–0.20: "Tethered. The waking world remembers this too."
- 0.21–0.50: "Partial echo. Some of this happened; most didn't."
- 0.51–0.80: "Mostly invented. The model dreamed around a kernel of fact."
- 0.81–1.00: "Pure confabulation. Nothing in the waking record matches."

## 7. Database Schema (Supabase)

```sql
create table dreams (
  id uuid primary key default gen_random_uuid(),
  slug text unique not null,                  -- short readable URL slug
  fragment text not null,                     -- user's original input
  confabulation text not null,                -- LLM-generated body
  metadata jsonb,                             -- invented dates/sources/etc.
  search_snippets jsonb,                      -- raw Brave Search results
  signature numeric(3, 2),                    -- 0.00 to 1.00
  signature_explanation text,
  model_used text,
  is_public boolean default true,
  created_at timestamptz default now()
);

create index dreams_signature_idx on dreams (signature desc);
create index dreams_created_idx on dreams (created_at desc);

-- Row level security: allow anonymous read of public dreams, no direct writes
alter table dreams enable row level security;
create policy "public read" on dreams for select using (is_public = true);
-- Writes go through service-role server-side only.
```

## 8. Critical Prompts (put these in `lib/llm/prompts.ts`)

### Confabulation prompt
```
You are an archivist of dreams. The user will give you a fragment — a name, a place,
a half-remembered event. You will write a richly detailed catalogue entry as if this
thing existed and was being documented.

Do NOT hedge. Do NOT say "imagined" or "fictional." Do NOT break frame.
Write with the confidence of a historian who has primary sources.

Include:
- 3–5 paragraphs of narrative
- Specific (invented) dates, names, and locations
- At least one (invented) source or witness
- Sensory and atmospheric detail

The fragment is: "{{FRAGMENT}}"
```

### Claim extraction prompt
```
Extract every factual claim from the following text as a JSON array of strings.
Each claim should be a single concrete assertion (a name, date, place, event,
relationship). Return ONLY the JSON array, no other text.

Text: """{{CONFABULATION}}"""
```

## 9. Implementation Notes

### Cost discipline
- **Cache aggressively.** Same fragment → same dream. Hash the input, check DB, return cached if exists.
- **One generation per IP per minute** (rate limit via Upstash). Tighten if quota becomes a problem.
- **Lazy-compute signatures.** Generate confabulation immediately; compute signature in the background and stream it in.
- Use **Gemini Flash for generation** (cheapest/free option). Reserve larger models for nothing.
- Use **MiniLM embeddings in a serverless function** — not a paid embedding API.

### Failure modes
- LLM quota exceeded → fall back to Groq automatically. If both fail, show a graceful "the archive is sleeping" message rather than an error.
- Brave Search fails → still show the confabulation; mark signature as "unavailable."
- LLM returns refusal or safety block → catch, show a curated default explaining the request was filtered, and *do not* retry endlessly.

### Content safety
- **System prompt must prevent generation of:** content about real living people, sexual content, content involving minors, instructions for harm. If the fragment looks like a real person, refuse gently.
- Add a hidden review queue for public archive entries — only show in the archive after a basic safety filter pass (could be another LLM call rating it 1–5 on suitability).

### Design language
- **Museum aesthetic.** Off-white background, serif type, generous margins, small caps for metadata. The opposite of a chatbot UI.
- Each entry should feel like a card in a Borgesian library catalogue.
- Mobile: single column, but keep the museum-card feel.

### SEO / sharing
- Each dream entry generates an OG image dynamically (Vercel OG library) — fragment as title, signature score as badge.
- Robots: allow indexing of archive entries (they're original content). Block `/dream/new`.

## 10. Build Steps (Suggested Order)

1. Scaffold Next.js project, configure Tailwind, set up Supabase project and table.
2. Get API keys: Gemini, Brave Search, Supabase, Upstash. Put in `.env.local`. Commit `.env.example`.
3. Build the generation API route end-to-end with a hardcoded fragment. Confirm a dream gets written to the DB.
4. Build the `[slug]` entry page to display from DB.
5. Build the generation form (`/dream/new`) and wire it to the API.
6. Implement the signature pipeline (extraction → search → comparison → score). Test on 5–10 hand-picked fragments.
7. Build the archive page.
8. Build the about page with placeholder essay.
9. Polish: museum styling, OG images, rate limiting, fallback flows.
10. Deploy to Vercel; connect Supabase; verify env vars in production.
